### 0.2 执行经验复盘

**403 / 登录权限 / 网络出口**

问题做法：
- 先把 403 当成 FastAPI 业务接口错误，容易误导排查方向。
- 只解释 HTTP 状态码，不结合截图里的关键错误短语和运行环境。

正确做法：
- 先问清 403 出现位置：本地接口、VS Code 扩展、Codex 登录、远程服务器还是浏览器 OAuth。
- 如果截图或日志出现 `token_exchange_failed`、地区/网络出口相关提示，应优先归因到账号权限、服务可用地区、代理或服务器出口，而不是项目代码。
- 涉及网络与代理时继续遵循“说明作用范围”：项目级、仓库级、用户级、机器级分别讲清楚。

适用场景：
- Codex 登录失败、VS Code Remote / SSH 环境连接 Codex、API 或扩展 403、账号授权异常。

**VS Code 项目级设置**

问题做法：
- 修改 `.vscode/settings.json` 时补丁上下文过大，遇到 BOM、换行或已有 JSON 细节时容易失败。
- 没说明设置是项目级还是用户级，用户不容易判断影响范围。

正确做法：
- 先读取真实 JSON，再用最小上下文补丁；补丁不匹配时缩小上下文并保留原有配置。
- 修改后读回验证 JSON，并说明 `.vscode/settings.json` 是当前工作区级设置。

适用场景：
- 关闭 VS Code Audio Cues、工作区 Python/conda 配置、项目级编辑器行为调整。

**FastAPI / MySQL / SQLAlchemy 学习**

问题做法：
- 为了让示例“更容易跑”，擅自把 MySQL 学习代码改成 SQLite，会偏离用户正在学习的目标。
- 把 Database Client 插件、MySQL Server、Python ORM 三件事混在一起讲，用户会不知道当前该点哪里、运行哪个文件。

正确做法：
- 学习场景下优先保持用户当前技术路线；需要临时替代方案时明确说“这是临时验证，不替代 MySQL 学习主线”。
- 把操作拆成三层：本机 MySQL Server 是否启动、Database Client 是否能连接、FastAPI/SQLAlchemy 脚本是否能读配置并访问数据库。
- SQL 练习脚本可以独立新增，不强行塞进 Python 文件；真实连接串继续放 `config/.env`，不提交。

适用场景：
- FastAPI ORM 入门、MySQL 安装向导、Database Client 连接、Python 学习脚本调试。

**论文阅读 skill 与 PDF/Word 产物**

问题做法：
- 在没有确认 skill 指令、PDF 路径、页数和标题前就生成分析，容易造成输出目录混乱或报告命名不稳定。
- 把中间解析文本、图片、公式、翻译缓存和正式报告混在一起，后续难复查。
- 用外部工具前不先检查依赖，遇到 `pdftotext`、LibreOffice、python-docx、python-pptx 等缺失时才中断。

正确做法：
- 对 `03_我创建的Skills/` 内的 skill，先读 `SKILL.md` 和直接引用的必要 reference，再执行。
- 论文任务先确认 PDF 路径、页数、标题、输出目录；快速阅读默认产出少而稳定的 4 个 Markdown。
- PDF/Word/PPT 处理前先检查本地依赖；缺工具时选择可用替代路径，并在最终说明里标注限制。
- 中间产物归入 `_cache/`、`assets/` 或明确的输出目录；正式入口文件保持少而清晰。

适用场景：
- paper-reading-fast、paper-reading-pipeline、Zotero PDF 翻译、中文 Word 生成、公式/图片抽取。

**科研 PPT / 组会材料**

问题做法：
- 直接把提示词工程内容搬进通信/通感算方向，容易显得空泛，缺少研究可落地性。
- 在用户要求“先捋思路给我审核”时直接生成文件，会浪费迭代成本。
- 只生成 PPT，不附来源推导、参考论文和终检图片，后续汇报前难检查。

正确做法：

- 用户要求审核时先给结构、论证链和页面规划；确认后再生成 PPT/Word。
- 组会 PPT 默认配套：PPTX、讲稿、来源与推导报告、参考论文索引、终检 PNG。
- 联网查论文时优先保留题名、年份、任务关系和用途线索，不把长论文原文塞进汇报材料。

适用场景：
- 研究生组会 PPT、提示词工程迁移到 ISAC/MEC/6G、论文依据驱动的选题论证。

适用场景：

## 2. 执行经验复盘

### 2.1 Git / GitHub / 网络配置

问题做法：
- 看到 VS Code Git 弹窗就直接当作仓库损坏或推送失败。
- 在混杂工作区里宽泛使用 `git add -A`，容易把缓存、日志、数据集或无关文件带入提交。
- GitHub 网络失败时没有说明代理配置作用范围，容易让用户担心电脑网络被改。
- 对被 `.gitignore` 忽略的文件一律改 ignore 规则，反而扩大配置影响。

正确做法：
- Git 操作前先查 `git status --short --branch`，必要时补查 `git log --oneline --decorate --max-count=5`、`git rev-parse HEAD`、`git reflog`。
- 如果 `HEAD`、`origin/main` 已对齐，VS Code `cannot lock ref 'HEAD'` 更可能是 Source Control stale 状态，先刷新 Source Control 或 Reload Window。
- 提交前定向 stage 文件；用户明确要上传 ignored 路径时，用 `git add -f <path>`，并说明这是一次性强制加入。
- GitHub 连接失败时先 `Test-NetConnection github.com -Port 443`；如需代理，优先 repo-local `.git/config`，并明确不是系统网络设置。
- 大文件上传前先量尺寸：浏览器上传受 25 MiB 单文件限制，GitHub 仓库单文件通常不能超过 100 MiB。

适用场景：
- GitHub 上传、VS Code Git 弹窗、仓库同步、数据集提交、网络代理诊断。

### 2.2 README / Markdown / 内容系统

问题做法：
- 不读真实 README 就套通用模板。
- 中文 Markdown 用 PowerShell 默认编码读取，导致乱码后误判内容。
- skill 检索无结果后继续耗时间。
- 资料包文件名与 README 文案没有对齐，容易出现“文档说一个名，磁盘是另一个名”。

正确做法：
- 先用 UTF-8 读取真实文件，再基于现有栏目重排。
- README 资源型页面优先整理为：总览、目录或分区、每节说明、统一资源链接、来源或许可说明。
- 链接用可读的 Markdown 列表呈现，不把大段原始 URL 堆在段落里。
- skill 找不到时尽快转为 GitHub Markdown 通用规范。
- 文档更新后对照真实磁盘文件名、大小和 Git 状态，保证本地与远程一致。

适用场景：
- README 重构、学习资料入口维护、中文 Markdown 整理、资料包同步。

### 2.3 Windows 文件整理

问题做法：
- 大规模移动/重命名前没有先总结影响范围。
- 脚本中断或用户改口后直接重跑，容易叠加旧状态。
- 只看终端换行输出，不生成报告，后续难审计。
- 用户已经给了快捷方式路径，却先大范围搜索安装目录。

正确做法：
- 有副作用前先确认路径、目标清单和步骤；用户要求“开始前先总结”时必须先列步骤。
- 中断后重新检查当前目录状态，再只处理剩余工作。
- 批量整理默认按人名/对象分组，保留状态标签：`（空）`、`（无封面）`、`（文件数）`。
- 批量移动和重命名后生成报告或映射文件。
- 用户提供应用快捷方式时优先使用该路径。

适用场景：
- 桌面资料整理、学生文件归档、花名册核对、下载文件二次分拣。

### 2.4 VS Code / 扩展 / CLI 排查

问题做法：
- 把“扩展已安装”和“命令行可运行”混成一个结论。
- 对 fork、CLI、Git 安装等术语解释过于抽象。

正确做法：
- 分开汇报：VS Code 扩展是否安装、终端 CLI 是否在 PATH、配置目录是否存在、版本命令是否能运行。
- 工具术语先贴近用户当前界面或命令解释，再给一个小例子。
- Windows 安装 Git 时说明 VS Code 使用系统 Git；普通 Intel/AMD Windows 默认推荐 Git for Windows x64 Standalone Installer。

适用场景：
- VS Code 工具链配置、Codex/Claude Code 扩展排查、Git 安装、术语解释。

### 2.5 Skill 检索与学习路线

问题做法：
- 多个 `npx skills find` 共用默认缓存路径，容易在 Windows 出现临时目录、权限或模块冲突。
- 只用精确关键词检索，容易因为 registry 命名不同而误判没有相关 skill。
- skill 发现类回答只给单个结论，用户缺少比较空间。

正确做法：
- 用户要求查找 skill 时，先读取相关 `SKILL.md`，再检索。
- Windows 下每个主题使用独立 repo-local cache，例如 `.npm-cache-agent-planning`。
- 本地 registry 无结果时，尽快扩展到相邻概念，并补充 skills.sh、GitHub 等候选来源。
- 对候选 skill 比较特点、适用场景和安装方式，让用户决定下一步。

适用场景：
- skill 发现、能力扩展、agent 工作流学习、学习路线规划。

### 2.6 Python / FastAPI / 数据库学习代码

问题做法：
- 把数据库连接串、密钥或本地配置直接写进源码或提交进仓库。
- 解释 Python/FastAPI 代码时只讲抽象概念，不结合当前文件。
- 运行测试前不看项目结构，容易跑过大的无关验证。

正确做法：
- 数据库连接串放在 `config/.env` 等本地配置文件中，并用 `.gitignore` 保护 `config/*.local`、`config/*.secret`、`config/.env`。
- 学习示例可以从 repo-local `.env` 读取配置，但提交时只提交代码和 ignore 规则，不提交真实密钥。
- 解释 FastAPI、SQLAlchemy、PyTorch 代码时结合当前文件和错误信息，给最小可运行或最小验证建议。

适用场景：
- FastAPI ORM 示例、MySQL 连接配置、Python 学习代码提交、密钥保护。

### 3.3 UI 与界面体验偏好

本周没有独立前端 UI 设计任务，因此这里主要从工具界面、文档界面和错误弹窗偏好归纳：

- 工具状态要拆开显示，例如“扩展已安装”和“CLI 可运行”分别说清楚。
- 报错界面应先降低不确定性：说明严重程度、是否已同步、下一步做什么。
- 文档界面偏好层级清晰、链接可读、来源清楚，不喜欢原始 URL 堆叠。
- 对工具型界面/工作流，偏好安静、可扫描、状态明确、操作路径短。

### 3.4 内容系统偏好

- README 更适合作为学习资源索引，而不是杂糅的链接堆。
- 中文内容优先保证 UTF-8 正确读取与写入。
- 说明文档保留来源线索，但不复制大段原文。
- 会话复盘要按任务类型沉淀经验，去重合并规则，避免每次从零解释。
- 规则要能直接指导后续 Codex 行为，少写口号，多写可执行动作。

## 4. 后续 Codex 可复用规则清单

1. 在本工作区回答配置、Git、VS Code、README、文件整理问题前，先检查真实本地状态。
2. 用户发截图或问“这是什么问题”时，先给一句清晰诊断，再给必要检查和解决步骤。
3. 涉及副作用操作时，先说明会改哪里；用户要求先总结时，必须先列步骤再执行。
4. Windows 文件批处理要先验证路径和当前状态，中断后重新检查，再继续。
5. 文件归档默认按人名/对象分组，并保留可见状态标签和文件数；批量移动或重命名后输出报告。
6. Git/GitHub 任务先查 `git status --short --branch`，提交前避免宽泛 `git add -A`。
7. 被 `.gitignore` 忽略但用户明确要上传的路径，说明原因后使用 `git add -f <path>`。
8. GitHub 网络失败时，先区分直连、代理、repo-local Git config 和系统网络设置。
9. VS Code Git 弹窗先查 `HEAD`、`origin/main`、`reflog`，不要直接建议 reset 或修历史。
10. 检查扩展配置时，分开汇报“扩展安装状态”和“终端 CLI 可用状态”。
11. skill 发现任务先读对应 skill 指南；无结果时扩展关键词并比较候选项。
12. Windows 上多次 `npx skills find` 使用独立 repo-local cache，避免默认缓存冲突。
13. 中文 README 或文档读取优先指定 UTF-8。
14. README 重构先读实际内容，结构建议要贴合现有栏目，不套通用模板。
15. 复杂学习路线用阶段式大纲，每阶段给练习和进入下一阶段的检查问题。
16. Python/FastAPI/数据库示例不要提交真实 `.env`、密钥或本地连接串。
17. 会话复盘只读取摘要、索引、元数据和相关片段，不整文件载入大型 session 日志。
18. 维护 AGENTS.md 时去重合并相近规则，并保留日期范围或任务类型作为来源线索。

## 5. AGENTS.md 维护建议

本仓库已经存在项目级 `AGENTS.md`，适合继续维护以下长期规则：

- 默认中文沟通、先诊断后步骤、术语贴近当前界面或命令。
- 修改或解释文件前先读取真实内容，中文文件优先 UTF-8。
- Git 操作前先查状态，提交前定向 staged，避免把缓存、日志、数据集和密钥带入提交。
- GitHub 网络或代理配置必须说明作用范围，优先 repo-local。
- Windows 文件整理任务先确认路径、总结步骤、输出报告，并在中断后重新检查。
- Skill 检索使用独立 `.npm-cache-*`，无结果时扩大关键词并比较候选。
- FastAPI / 数据库学习代码使用本地配置文件存放连接串，真实 `.env` 不进仓库。
- 会话复盘任务优先使用记忆索引和 rollout summary，不复现原始日志或隐私内容。

适合考虑加入用户级 AGENTS.md 的跨项目规则：

- 中文偏好、plain-language 诊断、命令逐行解释。
- 扩展安装状态与 CLI 可用状态分开汇报。
- 高副作用任务先说明影响范围。
- 复杂学习路线使用阶段目标、练习和检查问题。
- 会话日志复盘时只使用摘要/索引/片段，不复现隐私或长原文。

是否加入用户级 AGENTS.md 需要每次单独询问；本次先更新项目级 `AGENTS.md`，不擅自改用户级全局规则。
