# Codex 从需求到交付完整链路使用文档

资料来源：`Claude Code 学习手册/books/Codex橙皮书.pdf` 第四篇“标准工作流：从需求到交付的完整链路”，并结合本仓库 `AGENTS.md` 的项目规则整理。  
适用场景：功能开发、Bug 修复、UI 优化、README/文档整理、学习代码改造、FastAPI/Python 小项目维护。

## 0. 核心原则

不要把需求直接变成代码。更稳定的做法是让 Codex 按固定链路推进：

1. 需求拆解：先理解目标、边界、文件范围和风险。
2. 制定计划：先说准备怎么做，确认后再改。
3. 小步实现：一次只改一个功能点，避免顺手重构。
4. 测试验证：用命令、页面、流程证明修改有效。
5. 代码审查：看 diff、风险、边界条件和是否误伤旧功能。
6. 提交与复盘：生成 commit/PR 说明，把经验沉淀到文档或 `AGENTS.md`。

一句话记忆：AI 负责执行，人负责拍板；每一步都要可验证、可回退、可复盘。

## 1. 第一步：需求拆解

目标：让 Codex 在动手前先弄清楚“为什么做、做什么、不做什么、怎么验收”。

适合使用的情况：

- 需求比较模糊，例如“优化首页”“修一下登录”“整理一下项目”。
- 你还不确定涉及哪些文件。
- 你担心 Codex 改动范围太大。
- 任务涉及 UI、接口、鉴权、数据库、依赖、全局样式等可能影响其他功能的内容。

必须拆清楚的内容：

- 背景：当前项目是什么，为什么要做这个需求。
- 类型：新功能、Bug 修复、UI 优化、重构、文档整理，还是环境配置。
- 目标：本次具体解决到什么程度。
- 非目标：哪些内容不属于本次范围。
- 文件：哪些文件可能相关，哪些文件不能动。
- 验收：功能、页面、代码、测试分别怎样算完成。
- 风险：影响范围、样式污染、过度重构、新增依赖、旧功能回归。

可复制提示词：

```markdown
请先帮我做需求拆解，不要立刻修改代码。

需求：
【这里写你的需求】

请按下面结构分析：

## 1. 背景是什么

- 当前项目大概是什么
- 为什么要做这个需求
- 这个需求属于新功能、Bug 修复、UI 优化，还是重构

## 2. 要解决什么问题

- 当前具体问题是什么
- 本次要解决到什么程度
- 哪些内容不是本次范围

## 3. 哪些文件可能相关

- 请根据项目结构判断可能涉及哪些文件
- 先列出来，不要直接修改

## 4. 哪些功能不能动

- 不要改哪些逻辑
- 不要动哪些接口
- 不要影响哪些页面或组件

## 5. 什么结果算完成

- 功能完成标准
- 页面完成标准
- 代码完成标准

## 6. 需要哪些测试

- 需要运行什么命令
- 需要手动检查哪些页面
- 需要重点验证哪些流程

## 7. 有哪些风险

- 可能影响哪些功能
- 是否有样式污染风险
- 是否有重构过度风险
- 是否有新增依赖风险

最后，请给我一个简短的执行建议：

- 建议先做哪一步
- 是否需要我确认后再修改
```

本仓库额外规则：

- 修改或解释文件前，先读取真实文件内容。
- 中文 Markdown、README、配置文件优先按 UTF-8 读取。
- 对 FastAPI、PyTorch、Python 学习代码，要结合当前文件解释，不要只讲抽象概念。

## 2. 第二步：制定计划

目标：让 Codex 先输出准备做什么、改哪些文件、怎么验证，用户确认后再执行。

计划应该包含：

- 任务拆分：分几步做，每一步解决什么。
- 文件范围：每一步预计涉及哪些文件。
- 修改策略：新增、修改、删除、重构分别是什么。
- 风险控制：哪些地方需要保持不动。
- 测试方式：修改后跑什么命令、手动检查什么。
- 回退思路：如果方向不对，怎样尽量少回滚。

可复制提示词：

```markdown
先不要写代码，也不要修改任何文件。

请根据当前需求和项目结构制定修改计划。

请按下面结构输出：

1. 你对需求的理解
2. 准备分几步完成
3. 每一步要修改哪些文件
4. 每一步的预期结果
5. 哪些文件或逻辑不能动
6. 需要运行哪些测试或检查
7. 有哪些风险和不确定点

等我确认后，再开始执行。
```

判断计划是否合格：

- 是否只围绕当前需求。
- 是否明确“不改什么”。
- 是否有测试和验收标准。
- 是否把大任务拆成小任务。
- 是否说明不确定点，而不是替用户做业务决定。

## 3. 第三步：小步实现

目标：一次只实现一个功能点，降低出错概率，方便定位和回退。

小步实现的原则：

- 一次只改一个明确功能点。
- 只改和当前功能直接相关的文件。
- 不顺手重构无关代码。
- 不随意改目录结构、命名、依赖版本、全局配置。
- 发现额外优化点时，先记录为建议，不直接改。
- 遇到不确定业务逻辑、是否删除旧代码、是否新增依赖时，先停下来问。

可复制提示词：

```markdown
请开始小步实现。

当前只执行第【1】步：
【这里写本次只做的一个功能点】

要求：

1. 一次只改这个功能点
2. 只修改和当前功能直接相关的文件
3. 不要顺手重构无关代码
4. 不要修改目录结构
5. 不要新增不必要依赖
6. 不要删除已有功能
7. 不要改计划外的文件

修改完成后请停止，并输出：

1. 本次修改了哪些文件
2. 每个文件改了什么
3. 为什么这些修改是必要的
4. 有没有改到计划外内容
5. 有没有潜在风险
6. 下一步建议做什么

如果遇到不确定的地方，请先停下来问我，不要自行决定。
```

需要立刻暂停的信号：

- 改动文件数量突然变多。
- 删除了大量代码但没有解释。
- 新增了用户没要求的依赖。
- 修改了配置文件或全局样式。
- 改了和需求无关的页面、接口、组件。
- 代码风格和项目原有风格明显不一致。

## 4. 第四步：测试验证

目标：不要相信“我已经完成了”，而要用测试和检查证明它真的完成。

测试顺序建议：

1. 单元测试：如果项目已有测试，优先运行相关测试。
2. 类型检查：TypeScript 项目运行 typecheck；Python 项目可用语法检查或现有测试。
3. Lint：区分本次新增问题和项目历史问题。
4. 构建：前端项目运行 build，确认能打包。
5. 手动测试：按用户真实操作路径检查。
6. 浏览器测试：看页面、控制台、网络请求、移动端布局。
7. 回归测试：确认旧功能没有被影响。

可复制提示词：

```markdown
请对本次修改进行测试，不要继续新增功能。

请按下面顺序执行或说明：

1. 单元测试
   - 项目是否有单元测试
   - 如果有，请运行测试命令
   - 如果失败，请说明失败原因

2. 类型检查
   - 项目是否有 typecheck 命令
   - 如果有，请运行
   - 如果没有，请说明

3. lint
   - 运行 lint 检查
   - 区分本次新增问题和项目原有问题

4. 构建
   - 运行 build 命令
   - 如果失败，请说明报错原因和影响范围

5. 手动测试
   - 列出需要手动测试的页面
   - 列出用户操作步骤
   - 列出每一步预期结果

6. 浏览器测试
   - 检查页面显示
   - 检查控制台报错
   - 检查移动端布局
   - 检查关键按钮和交互

7. 回归测试
   - 检查本次修改是否影响旧功能
   - 列出可能受影响的页面、组件和流程

最后请输出测试总结：

- 哪些测试通过了
- 哪些测试失败了
- 失败原因是什么
- 是否可以进入下一步
- 是否需要先修复问题
```

本仓库测试注意：

- 运行测试前先看项目结构，不要盲目跑大范围命令。
- Python 学习脚本可以先做最小验证，例如语法检查、单文件运行、相关示例运行。
- 如果项目本身已有历史报错，要明确区分“历史问题”和“本次新增问题”。

## 5. 第五步：代码审查

目标：测试通过不等于可以交付，还要检查改动是否符合计划、是否有风险。

审查重点：

- 文件范围：是否只改了该改的文件。
- 修改原因：每个文件为什么要改。
- 删除内容：是否误删旧逻辑、fallback、兼容代码、配置。
- 边界条件：空数据、接口失败、未登录、权限不足、重复点击、移动端。
- 安全问题：密钥、token、账号密码、权限判断、输入校验、敏感信息。
- 业务逻辑：能跑不代表逻辑正确，要检查正常流程和异常流程。
- 风格一致性：是否符合原项目代码风格和结构。

可复制提示词：

```markdown
请对本次修改做代码审查，不要继续写代码。

请按下面结构审查：

## 1. Codex 自审

- 本次是否只改了计划内文件
- 是否有无关重构
- 是否有新增依赖
- 是否有硬编码
- 是否有误删旧逻辑

## 2. 修改范围审查

- 修改了哪些文件
- 每个文件为什么要改
- 是否存在计划外修改
- 是否有大面积无解释修改

## 3. 边界条件审查

- 空数据如何处理
- 接口失败如何处理
- 用户未登录如何处理
- 权限不足如何处理
- 重复点击如何处理
- 移动端是否可能异常

## 4. 安全问题审查

- 是否暴露密钥、token、账号密码
- 是否影响权限判断
- 是否缺少输入校验
- 是否可能泄露敏感信息
- 是否修改了接口鉴权逻辑

## 5. 删除内容审查

- 删除了哪些代码
- 删除原因是什么
- 是否确认没有其他地方依赖
- 是否可能影响旧功能

## 6. 业务逻辑审查

- 是否符合需求
- 正常流程是否正确
- 异常流程是否正确
- 是否影响旧业务规则
- 是否有不确定的业务假设

## 7. 审查结论

请最后给出结论：

- 可以继续
- 需要小修
- 需要回退部分修改
- 需要重新制定计划

注意：只审查，不要继续修改代码。如果发现问题，请先说明问题和建议，等我确认后再改。
```

高风险任务建议：

- 涉及登录、支付、权限、数据库、鉴权、上传、真实用户数据时，不要只依赖一次自审。
- 可以让另一个模型或另一次独立审查专门挑错，但最终决定仍由人做。

## 6. 第六步：提交与复盘

目标：把“代码能跑”变成“可以交付”，并把本次经验沉淀为下次可复用规则。

提交前检查：

- 是否只有计划内文件被修改。
- 是否测试和手动检查已经完成。
- 是否有未完成事项。
- 是否需要更新 README、`.env.example`、docs、CHANGELOG、`AGENTS.md`。
- 是否有需要人工确认的风险。

commit message 建议：

- `feat: 新增用户资料页`
- `fix: 修复登录后跳转异常`
- `style: 优化首页移动端布局`
- `docs: 更新项目使用说明`
- `refactor: 简化商品卡片组件`

PR 描述建议结构：

```markdown
## 本次修改

- 【说明本次做了什么】

## 修改原因

- 【说明为什么要改】

## 涉及文件

- 【列出关键文件和作用】

## 测试结果

- 【列出已运行的命令和手动测试结果】

## 风险说明

- 【说明可能影响哪里、需要重点看什么】
```

复盘要记录：

- 这次哪里需求没说清楚。
- 计划阶段漏掉了什么。
- Codex 是否顺手改了无关内容。
- 哪些测试遗漏了。
- 哪些提示词有效。
- 哪些规则应该加入 `AGENTS.md`。

可复制提示词：

```markdown
请进入提交与复盘阶段，不要继续新增功能。

请按下面结构输出：

## 1. Commit 建议

- 生成一个合适的 commit message
- 使用 conventional commit 格式
- 不要夸大本次修改范围

## 2. PR 描述

请生成 PR 内容，包括：

- 本次修改
- 修改原因
- 涉及文件
- 测试结果
- 风险说明
- reviewer 需要重点看的地方

## 3. 问题记录

请复盘本次过程：

- 遇到了哪些问题
- 原因是什么
- 如何解决
- 下次如何避免

## 4. Prompt 总结

请总结：

- 哪些 Prompt 有效
- 为什么有效
- 适合什么场景复用
- 是否建议加入 AGENTS.md

## 5. AGENTS.md 更新建议

请输出适合加入 AGENTS.md 的长期规则：

- 修改前规则
- 修改中规则
- 测试规则
- 提交规则

## 6. 项目文档更新建议

请判断是否需要更新：

- README.md
- .env.example
- docs/
- CHANGELOG.md
- 其他项目文档

最后请给出交付结论：

- 是否可以提交
- 是否可以发 PR
- 是否还有未完成事项
- 是否有需要人工确认的风险
```

## 7. 完整链路速查表

| 阶段 | 要问 Codex 什么 | 人要确认什么 | 产出 |
| --- | --- | --- | --- |
| 需求拆解 | 先分析需求，不要改代码 | 目标、边界、风险是否清楚 | 需求拆解报告 |
| 制定计划 | 先计划，确认后执行 | 文件范围和步骤是否合理 | 修改计划 |
| 小步实现 | 只做当前一步 | 是否超范围修改 | 小范围 diff |
| 测试验证 | 跑测试并列手动检查 | 结果是否可信 | 测试总结 |
| 代码审查 | 只审查，不继续改 | 是否有风险和误删 | 审查结论 |
| 提交复盘 | 生成 commit/PR/规则建议 | 是否可交付 | 提交说明和复盘 |

## 8. 本仓库推荐工作流

### 8.1 修改 Python / FastAPI 学习代码

1. 先读当前文件和相关 README。
2. 让 Codex 解释当前代码结构和目标。
3. 让 Codex 制定最小修改计划。
4. 一次只改一个函数、一个路由或一个示例点。
5. 运行最小验证，例如单文件运行、语法检查、相关接口启动检查。
6. 让 Codex 总结修改前后差异和学习要点。

### 8.2 修改 README / Markdown

1. 用 UTF-8 读取真实文件。
2. 先分析当前结构，不直接套模板。
3. 给出重排方案，让用户确认。
4. 改完后检查标题层级、链接、来源说明。
5. 避免把原始 URL 堆在段落里。

### 8.3 Git / GitHub 操作

1. 先运行 `git status --short --branch`。
2. 只 stage 用户目标相关文件。
3. 避免 `git add -A` 带入无关内容。
4. commit 前说明修改范围和测试结果。
5. 遇到 VS Code Git 弹窗，先查 `HEAD`、`origin/main`、`reflog`，不要直接重置。

### 8.4 文件整理和批量操作

1. 先确认路径和当前文件状态。
2. 先总结操作步骤和影响范围。
3. 移动或重命名前保留映射关系。
4. 中断后重新检查状态，再继续。
5. 输出报告，方便人工审计。

## 9. 常见翻车点与防护句

| 翻车点 | 防护句 |
| --- | --- |
| Codex 直接改代码 | “先不要写代码，也不要修改任何文件。” |
| 改动范围太大 | “一次只实现当前功能点，不要改计划外文件。” |
| 顺手重构 | “不要顺手重构无关代码，优化建议先记录。” |
| 新增依赖 | “不要新增不必要依赖；确实需要时先说明原因并等我确认。” |
| 删除旧逻辑 | “不要删除已有功能；如需删除，先说明依赖和影响范围。” |
| 测试失败后乱修 | “测试失败先总结原因和影响范围，不要直接继续修改。” |
| 只看功能能跑 | “请做代码审查，重点检查边界、安全、误删和业务逻辑。” |
| 经验没沉淀 | “请总结有效 Prompt，并建议是否加入 AGENTS.md。” |

## 10. 最小可复用总提示词

当你不知道怎么开始时，可以直接复制这段：

```markdown
请按 Codex 标准工作流处理这个需求：

需求：
【写你的需求】

要求：

1. 先做需求拆解，不要立刻修改代码。
2. 拆解后制定计划，列出可能涉及的文件、不能动的功能、测试方式和风险。
3. 等我确认后，再小步实现。
4. 每次只改一个功能点，不要顺手重构无关代码，不要新增不必要依赖。
5. 修改后先测试，再代码审查。
6. 最后给出 commit 建议、PR 描述、风险说明和复盘建议。

如果遇到不确定的业务逻辑、是否删除旧代码、是否新增依赖、是否超出计划修改，请先停下来问我。
```
